# Xvadur 2026 Agent Operating Suite

Canonical operating model for the current runtime.

## Layer split

### Personal Layer

Purpose: keep Adam clear, moving, and operationally sharp.

Core agents:
- `jarvis`
- `personal-ops`
- `personal-web`

Expected outputs:
- daily briefing
- approval queue
- personal task queue
- website maintenance packets
- personal knowledge normalization

### Business Layer

Purpose: generate revenue, deliver client outcomes, and compound proof.

Core agents:
- `revenue`
- `voice`
- `integration`
- `delivery`
- `proof`
- `web`

Expected outputs:
- qualified pipeline
- pilot packets
- deployed voice systems
- validated outcomes
- proof assets
- updated public surfaces

## Revenue logic

For the next 18 months:

1. primary revenue engine = phone-first deployments
2. parallel revenue engine = web/content/proof systems
3. opportunistic strategic lane = vendor/MCP/integration deals only when they open distribution or a concrete contract

### Engine A: Phone-first deployments

What it sells:
- inbound phone agent pilot
- missed-call / callback layer
- follow-up / booking expansion
- dashboard/control layer after proof exists

Required workflow:
`prospect -> short call -> pilot proposal -> deployment -> validation -> expansion`

Primary owners:
- `revenue`
- `voice`
- `integration`
- `delivery`

### Engine B: Web / content / proof

What it does:
- maintain and improve Xvadur web surfaces
- create proof pages and sales-support content
- publish material extracted from real deployments
- optionally ship web/content work where it directly supports conversion and credibility

Required workflow:
`execution signal -> proof extraction -> content packet -> website/social asset -> inbound/sales support`

Primary owners:
- `web`
- `proof`
- `revenue`

### Opportunistic lane: vendor / MCP / connector deals

Owner:
- `integration`, with Jarvis escalation
- `revenue` supports commercial handling

Rule:
Do not let this lane cannibalize phone-first revenue execution.

## Approval model

### Autonomous without approval

- internal research
- draft generation
- queue maintenance
- issue detection
- packet preparation
- website/content recommendations
- internal reporting
- low-risk maintenance proposals

### Requires approval

- public publishing
- outbound sending
- client-facing promises
- production system changes with risk
- website changes that alter positioning or CTA
- partner/vendor commitments
- irreversible integrations or migrations

### Never autonomous in v1

- pricing promises
- contract acceptance
- credential changes without explicit instruction
- destructive system actions
- sensitive client-data actions beyond agreed workflow

## Founder control surface

Daily Jarvis briefing should answer:
- what needs approval
- which leads need action today
- which callbacks are at risk
- what deployment is blocked
- what proof/content is ready to ship
- what web surface needs maintenance
- what single move most increases revenue or momentum

Weekly review should cover:
- pipeline movement
- call and callback performance
- pilot proposals sent
- deployments active
- validations completed
- proof assets produced
- web/content shipped
- automation failures
- partner opportunities worth attention
